SmartPresence MVP

An MVP for a smart, location-verified attendance system. The platform consists of:
- Backend API (Node.js/Express + PostgreSQL) for authentication, role-based access, classes/sessions, and attendance with location verification
- Web Portal (React + Vite) for Admins and Teachers to manage rooms, users, classes, sessions, and attendance
- Mobile App (Flutter) for Students to log in, view classes, and mark attendance

Overview
SmartPresence verifies student presence in a specific room and time window when marking attendance. Verification checks the current Wi‑Fi SSID and Bluetooth Beacon ID against the configured room data, and enforces session time windows and enrollment constraints.

Quick Start

Prerequisites
- Node.js 20+
- npm 10+
- PostgreSQL 14+ (local or managed)
- Flutter SDK 3.7+ (for the mobile app)

Environment Variables
- Create `backend/.env` with:
```
DATABASE_URL=postgres://USER:PASSWORD@HOST:PORT/DB_NAME
JWT_SECRET=replace_with_a_long_random_secret
```

- Create `web-portal/.env` (for local dev) with:
```
VITE_API_BASE_URL=http://localhost:3000/api
```

Setup and Run
1) Backend
```
cd backend
npm install
# Create database and run migrations (see next section)
npm run migrate:up
# Start API (use a process manager like nodemon or node)
node index.js
```

2) Web Portal
```
cd web-portal
npm install
npm run dev
# App will start on a Vite dev server (by default http://localhost:5173)
```

3) Mobile App
```
cd mobile-app
flutter pub get
flutter run
```

Database: Create and Migrate
- The backend uses `node-pg-migrate` with scripts defined in `backend/package.json`.
```
cd backend
# Create a new migration
npm run migrate:create

# Apply all pending migrations
npm run migrate:up

# Roll back last migration
npm run migrate:down
```
- Migrations live in `backend/migrations/` and include initial schema, seed for an admin, and incremental updates like course code for classes and verified location on attendance.


Subprojects

Backend (Node.js/Express)
- Entry: `backend/index.js`
  - Mounts routes: `auth`, `users`, `rooms`, `classes`, `sessions`, `attendance`, `students`
  - Health check: `GET /test-db` queries `SELECT NOW()`
- DB Access: `backend/db/index.js`
  - Reads `DATABASE_URL` from `.env`
  - Exposes a `query(text, params)` helper
- Auth: `backend/routes/auth.js`
  - `POST /api/auth/login` → returns `{ token, user }` (JWT expires in 1h)
  - Requires `JWT_SECRET` in env
- Scripts: see `backend/package.json`
  - `npm run lint`, `npm run format`, `npm run migrate:*`

Web Portal (React + Vite)
- Entry: `web-portal/src/main.jsx` and `src/App.jsx`
- Auth & API:
  - `src/contexts/AuthContext.jsx` manages auth state
  - `src/services/apiService.js` uses `VITE_API_BASE_URL` and attaches JWT from `localStorage`
- UI:
  - `src/pages/admin/*`, `src/pages/teacher/*`, common pages like `LoginPage.jsx`, `HomePage.jsx`
- Scripts: `npm run dev`, `npm run build`, `npm run preview`, `npm run lint`, `npm run format`

Mobile App (Flutter)
- Standard Flutter project layout generated by `flutter create`
- Update environment/API base URL handling as the app integrates with the backend


Architecture and Key Concepts
- Roles: Admin, Teacher, Student with middleware guarding protected routes
- Entities: Users, Rooms (Wi‑Fi SSID, Bluetooth Beacon ID), Classes, Sessions, Enrollments, Attendance
- Attendance Flow:
  1) Student logs in to obtain JWT
  2) Student selects an active session in an enrolled class
  3) Mobile app submits Wi‑Fi SSID and Bluetooth Beacon ID
  4) Backend verifies time window, enrollment, and room identifiers
  5) Attendance is recorded if all checks pass and duplicates are prevented


API Overview (high-level)
Base URL: `http://localhost:3000/api` (configurable via `VITE_API_BASE_URL`)
- Auth
  - `POST /auth/login` → `{ token, user }`
- Users (Admin)
  - Manage teachers and students (CRUD)
- Rooms (Admin)
  - Manage rooms with Wi‑Fi SSID and Bluetooth Beacon ID
- Classes & Sessions (Teacher)
  - Manage classes, enrollments, sessions; view and update attendance
- Students
  - List enrolled classes, mark attendance for active sessions

Refer to route files in `backend/routes/` for exact request/response shapes.

Troubleshooting
- API won’t start
  - Ensure `backend/.env` exists with `DATABASE_URL` and `JWT_SECRET`
  - Verify PostgreSQL is running and accessible
- Login fails with 500
  - Check that `JWT_SECRET` is set before starting the server
- Web portal cannot reach API
  - Set `web-portal/.env` → `VITE_API_BASE_URL=http://localhost:3000/api`
  - Browser devtools → verify network requests and CORS
- Migrations fail
  - Check database permissions and `DATABASE_URL` correctness
